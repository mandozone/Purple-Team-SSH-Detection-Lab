Project Overview

The complete attack-detect-respond lifecycle, on real evidence.

This lab demonstrates the full life of an SSH credential-access attack: configuring the target service, mapping the attack surface with Nmap, capturing live traffic with tcpdump, ingesting /var/log/auth.log into Splunk, building threshold-based SPL detections tagged with MITRE ATT&CK metadata, and deploying fail2ban as a verified defensive control.

Every screenshot in this write-up was generated from live activity inside an isolated VirtualBox Kali Linux environment — not a sample dataset. The lab follows a purple-team approach: the attack and the detection are both performed in the same controlled environment, so each detection rule can be evidenced against the exact behavior that triggered it.

Stage 1 — Service Setup

SSH service configuration & verification.

The first step is confirming the target SSH service is active and accepting connections. systemctl is used to verify state — rather than assuming the service is running — which reflects standard SOC and sysadmin practice.

sudo systemctl enable ssh    # Enable SSH to start on boot
sudo systemctl start  ssh    # Start the SSH service
sudo systemctl status ssh    # Verify active (running) state

Stage 2 — Reconnaissance

Mapping the attack surface with Nmap.

With the service confirmed, Nmap was used to verify what the attacker actually sees. This mirrors the recon phase an adversary performs before launching an authentication attack — identifying open services and entry points.

PortStateServiceSecurity implication
22/tcpOPENSSHPrimary brute-force target — repeated auth attempts possible
80/tcpOPENHTTPSecondary attack surface — web-based attacks possible
All othersFILTERED—998 ports blocked — firewall reducing overall exposure

🧠 MITRE ATT&CK Context — T1046: Network Service Discovery

This Nmap scan maps to T1046 (Network Service Discovery), an adversary technique used to identify open ports and services prior to exploitation. In a real intrusion, T1046 immediately precedes the T1110 brute-force attempt — catching the attacker at recon is the ideal SOC outcome.

Stage 3 — Packet Capture

Ground-truth evidence with tcpdump.

Packet capture provides ground-truth evidence of what actually traversed the network — independent of application logs, which can be tampered with or suppressed. tcpdump was run before and during the attack to capture the SSH handshake, banner exchange, and packet timing.

⚠️ Important Distinction — Brute-Force vs. DoS

The capture shows repeated SSH connection attempts, not a SYN flood. A SYN flood (T1498) sends incomplete handshakes to exhaust server resources. What is observed here is T1110.001 (Password Guessing) — every connection completes the TCP handshake and then attempts SSH authentication. The goal is credential access, not service disruption.

sudo tcpdump -ni any port 22 -ttt   # Capture SSH traffic, relative timestamps
sudo tcpdump -ni any port 80 -ttt   # Capture HTTP traffic

# Flags:
#  -n     no DNS resolution (show raw IPs)
#  -i any capture on all interfaces
#  -ttt   show relative timestamp between packets

Stage 5 — Detection Engineering

Catching T1110.001 in Splunk with SPL.

With /var/log/auth.log ingested into the main index, two SPL queries do the heavy lifting: a baseline search to confirm the attack is present, and a threshold-based, MITRE-tagged search that distinguishes scripted brute-force from a user mistyping a password.

Detection Query — Baseline

index="main" ssh source="/var/log/auth.log" "authentication failures"

# Result: 16+ events in initial window
# Source IP : rhost=10.10.14.45 (anonymized)
# Target    : user=kali
# Log path  : /var/log/auth.log (live ingestion)

This query searches the main index for events from /var/log/auth.log containing the phrase "authentication failures" — the exact string written by the Linux PAM subsystem when SSH login attempts fail. Filtering on ssh scopes results to SSH-specific failures rather than other PAM-managed services.

Detection Query — Advanced (Threshold-Based, MITRE-Tagged)

index=main source="/var/log/auth.log" "authentication failures"
| bucket _time span=1m
| stats count by _time rhost user
| where count > 10
| sort -count
| table _time rhost user count
| eval mitre_technique    = "T1110 - Brute Force"
| eval mitre_subtechnique = "T1110.001 - Password Guessing"
| eval mitre_tactic       = "Credential Access"
| eval severity = case(count>100,"HIGH", count>30,"MEDIUM", true(),"LOW")

Why the threshold works

bucket _time span=1m groups events into one-minute windows, then stats count by _time rhost user counts failures per source/user. where count > 10 filters out noise — a legitimate user mistyping a password rarely exceeds 3–5 failures per minute.

LOW · ≤30 MEDIUM · >30 HIGH · >100

Why MITRE tagging matters

The eval commands stamp each matching event with its MITRE tactic, technique, and sub-technique — making every alert framework-aware and directly actionable. Analysts triaging this alert see T1110.001 — Password Guessing and know exactly which response playbook to pull.

Attack visualisation & volume confirmation

🔥 Why 249 Failures Matters

A user mistyping their password: typically 2–5 failures before lockout or success. A misconfigured automation script: typically <20 failures before someone intervenes. An automated brute-force tool: exactly this pattern — sustained high volume, consistent timing, single source, single target. False-positive likelihood here is extremely low.

Stage 6 — Defensive Response

fail2ban — detection without response is incomplete.

Most lab write-ups end at "recommended mitigations". This project goes further by deploying and verifying fail2ban as a real defensive control inside the same environment where the attack was observed. The full attack-detect-respond cycle is closed.

Install & enable

sudo apt install fail2ban -y

sudo systemctl enable fail2ban
sudo systemctl start  fail2ban
sudo systemctl status fail2ban

# Confirm the SSH jail is active
sudo fail2ban-client status sshd

Verify the ban

sudo fail2ban-client status sshd

# Status for the jail: sshd
#    Currently banned: 1
#    IP list:          10.10.14.45

# Manual unban for continued lab testing
sudo fail2ban-client unban 10.10.14.45

How fail2ban kills T1110.001

BehaviorDefault settingEffect on brute-force
Monitors /var/log/auth.logBuilt-in SSH filterWatches the exact log source ingested into Splunk
Failure threshold (maxretry)5 failuresTriggers ban after 5 attempts — vs. 249 observed without it
Ban duration (bantime)10 minutesSource IP blocked from further connection attempts
Find window (findtime)10 minutesFailures counted within a rolling window

💡 Impact on this lab's attack

With fail2ban active, the 249-failure campaign would have been terminated after the 5th failure. The attacker's IP would have been banned for 10 minutes — halting the brute-force attempt before any meaningful password guessing could occur.

Response Planning

From IOC to incident-response playbook.

Immediate actions

  • Block 10.10.14.45 at the firewall — 249 failures with no legitimate business justification; fail2ban auto-ban should already be in effect.
  • Verify no successful logins occurred — search auth.log for "Accepted password" or "Accepted publickey" from the source IP before and after the failure window.
  • Audit the targeted account — check user kali for unauthorized SSH key additions, password changes, recent sudo activity, or new authorized_keys entries.
  • Review fail2ban ban log — confirm the IP was banned and document the trigger timestamp.

Secondary actions

  • Multi-account targeting check — stats count by user to confirm whether other accounts were targeted beyond kali.
  • Lateral movement check — verify the source IP did not attempt connections to other lab hosts.
  • Set a Splunk saved alert: count > 10 per source/user/minute → notification.
  • Harden SSH — consider PasswordAuthentication no in sshd_config in favour of key-based auth.

🚨 Escalate to Tier 2 if ANY of these are true

✔ Successful authentication detected from the source IP at any point ·  ✔ New SSH keys added to ~/.ssh/authorized_keys on any account ·  ✔ New user accounts created on the target system ·  ✔ Same source IP attacking multiple hosts on the network ·  ✔ Attack originates from inside the trusted network (not the isolated lab subnet).

Final SOC reporting statement

Activity observed from source IP 10.10.14.45 [anonymized] is consistent with MITRE ATT&CK T1110.001 (Password Guessing), a sub-technique of Brute Force under the Credential Access tactic. 249 authentication failures against user kali within the observation window, combined with automated libssh tooling fingerprinted in packet capture, confirm scripted credential-access behavior.

False-positive likelihood is extremely low given the sustained volume, timing consistency, and single-source targeting pattern observed across all evidence sources. Containment: source IP blocked via fail2ban. No successful authentication detected. Recommend SSH hardening (key-based auth only) as a permanent control.
